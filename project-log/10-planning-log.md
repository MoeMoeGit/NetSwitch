### ADR-001 [2026-05-22] 文档恢复口径：以当前代码实现为准

**状态**：已采用

**替代关系**：无

**背景与需求**：`project-log/` 曾被误删，恢复后需要把原始设计文案补入标准文档。原始设计与当前代码已有多处差异，例如 Python 版本、网卡检测方式、配置字段命名、托盘菜单和图标生成方式。

**采用的方案**：以当前代码实现为准补齐文档；与理想设计或原始设计不一致、且可能需要修复的内容，写入 `05-current-status.md` 和 `11-code-review-log.md`。

**备选方案**：

1. 完全按原始设计文案写文档
   - 优点：保留最初产品意图。
   - 缺点：会与当前代码冲突，后续 AI / 开发者容易按错误字段或错误依赖改代码。
   - 放弃原因：当前需求是恢复可用项目知识库，需要文档先可信。

2. 完全只描述代码，不记录差异
   - 优点：文档简洁。
   - 缺点：会丢失原始设计中的待优化意图，例如最多 5 秒轮询等待网络生效。
   - 放弃原因：用户明确要求把 bug 和优化点写入评审文档。

**决策依据**：

- 用户确认 Python 版本为 3.12。
- 用户确认网卡检测按当前实现写：`route print` + PowerShell。
- 用户确认配置字段按当前代码：`remark` / `ip_address` / `subnet_mask`。
- 用户确认首次导入方案名为「自定义」。
- 用户确认保留托盘菜单「开机自启」。
- 用户确认运行时图标按当前 Qt `QPixmap/QPainter` 写，Pillow 只用于构建期图标。
- 用户确认固定 `sleep(2)` 记录为待优化，应改为最多 5 秒、每 500ms 轮询。

**改动范围**：

- `project-log/00-project-overview.md`
- `project-log/01-function-design.md`
- `project-log/02-database-design.md`
- `project-log/03-api-design.md`

### ADR-002 [2026-05-22] 网络切换等待策略待改为轮询

**状态**：提议中

**替代关系**：无

**背景与需求**：当前 `network_controller.apply_profile()` 写入网络配置后固定 `time.sleep(2)`，随后读取网关并 ping。DHCP 获取、DNS 生效和网关更新耗时不稳定，固定等待过短会误判，过长又浪费时间。

**采用的方案**：后续改为最多 5 秒轮询，每 500ms 检测一次目标状态是否生效；一旦生效立即进入 ping 验证。

**备选方案**：

1. 保持固定 2 秒
   - 优点：实现简单。
   - 缺点：不稳定，慢 DHCP 场景容易误判。
   - 放弃原因：用户已确认应优化。

2. 固定等待 5 秒
   - 优点：比 2 秒更稳。
   - 缺点：多数场景无谓等待，体验变慢。
   - 放弃原因：轮询能兼顾速度和稳定性。

**决策依据**：

- 网络配置生效时间与网卡、DHCP 服务、系统负载有关。
- 工具目标是快速切换，同时不能牺牲可靠性。

**改动范围**：

- `network_controller.py`
- 相关测试或手动验证脚本
- `11-code-review-log.md` 修复状态
- `06-dev-log.md` 修复记录

### ADR-003 [2026-05-23] 已确认问题修复批次

**状态**：已采用

**替代关系**：无

**背景与需求**：用户要求先全面复核已发现问题，再修复确认成立的问题。本轮优先处理会影响真实使用可靠性的切换并发、单实例唤起、网关状态、失败回滚、自启反馈，以及低成本的数据一致性与文档问题。

**采用的方案**：

1. 切换并发：在网络写入层增加全局互斥锁，托盘和主界面入口在 UI 层尽量拦截重复触发。
2. 单实例唤起：主窗口设置固定标题，第二实例枚举隐藏窗口也可恢复已有主窗口。
3. 网关状态：配置应用成功但无法检测网关时返回 warning，而不是 success。
4. 回滚反馈：回滚命令逐条检查，回滚失败时把失败原因拼入 UI 错误消息。
5. 开机自启：注册表 Run 路径加引号，写入失败时不更新配置，并向用户提示。
6. 数据一致性：备用 DNS 增加校验；方案模式切换后清理不适用字段；配置文件损坏时先备份再重建。
7. 文档一致性：README 安装包构建命令改为生成后的 `installer_output.iss`。

**备选方案**：

1. 只在托盘入口加布尔锁
   - 优点：改动最小。
   - 缺点：主窗口和托盘之间仍可能并发写入网络配置。
   - 放弃原因：网络写入必须有底层互斥兜底。

2. 立即引入完整 IPC 单实例机制
   - 优点：更标准。
   - 缺点：改动范围更大，需要额外验证 QLocalServer / Windows 消息机制。
   - 放弃原因：当前主窗口隐藏后仍保留窗口句柄，修复窗口标题和隐藏窗口枚举即可覆盖现有问题。

**决策依据**：

- 网络写入是高风险操作，底层互斥优先于单一入口防抖。
- 本轮目标是修复确认问题，不引入新的网卡绑定、复杂队列或后台服务。
- 开机自启和回滚都不能向用户显示“假成功”。

**改动范围**：

- `main.py`
- `main_window.py`
- `network_controller.py`
- `profile_manager.py`
- `edit_dialog.py`
- `settings_dialog.py`
- `tray.py`
- `README.md`
- `project-log/05-current-status.md`
- `project-log/06-dev-log.md`

### ADR-004 [2026-05-24] 二次复核后的小范围修复批次

**状态**：已采用

**替代关系**：无

**背景与需求**：在已修复批次基础上，用户要求继续执行二次复核确认的问题，包括单实例隐藏窗口唤起、自启状态同步、子网掩码校验、配置保存原子性和 release 版本一致性。

**采用的方案**：

1. 单实例唤起：保留主窗口对象，不在关闭到托盘后清空引用，确保第二实例仍可恢复窗口。
2. 自启同步：启动时以注册表真实状态为准，并同步回配置文件，避免托盘和设置弹窗状态不一致。
3. 子网掩码校验：增加合法掩码校验，避免用户保存非法掩码。
4. 配置保存：改为临时文件写入后 `os.replace()` 原子替换，降低损坏概率。
5. Release 一致性：构建脚本在打包前校验 `VERSION` 与 Git tag 版本一致。

**备选方案**：

1. 继续依赖隐藏窗口枚举恢复
   - 优点：无需额外 IPC。
   - 缺点：窗口对象生命周期不稳定，唤起可靠性不足。
   - 放弃原因：现有模式下保留窗口引用代价更低。

2. 仅在 UI 表面同步自启状态
   - 优点：改动最小。
   - 缺点：配置文件和注册表仍可能不一致。
   - 放弃原因：用户看到的设置必须和真实系统状态一致。

**决策依据**：

- 这些问题都属于低改动成本、高收益的可靠性修复。
- 原子写和版本校验不会改变功能路径，但能减少发布和配置风险。

**改动范围**：

- `main.py`
- `edit_dialog.py`
- `network_controller.py`
- `profile_manager.py`
- `scripts/build.py`
- `project-log/05-current-status.md`
- `project-log/06-dev-log.md`

### ADR-005 [2026-05-24] 软件内更新检查与帮助反馈入口

**状态**：已采用

**替代关系**：无

**背景与需求**：用户在 Windows 端手动卸载、重新下载 Release 更新的流程过于繁琐，希望在软件内完成版本检查、更新获取，以及查看作者 / GitHub 主页 / Issue 入口。

**采用的方案**：

1. 在设置页增加“检查更新”按钮，手动触发 GitHub Release 查询。
2. 当发现新版本时，提供“下载并安装”和“打开发布页”两个动作。
3. 在设置页增加“帮助与反馈”区域，展示作者姓名和邮箱，并提供 GitHub 主页与 Issue 入口。
4. 更新采用 GitHub Release 作为分发源，不引入独立更新服务。

**备选方案**：

1. 启动时自动拉取更新并强制提示
   - 优点：更自动化。
   - 缺点：会在用户不需要时打扰，且更新体验更难控制。
   - 放弃原因：当前阶段先优先提供手动检查更新入口。

2. 使用专门的第三方自动更新框架
   - 优点：流程更完整。
   - 缺点：会引入额外依赖和平台适配成本。
   - 放弃原因：当前项目体量较小，GitHub Release 已足够承载更新分发。

**决策依据**：

- NetSwitch 是单机桌面工具，更新源固定，GitHub Release 足够直接。
- 用户更需要“少一步去网页找安装包”，而不是复杂的后台升级系统。
- GitHub 主页和 Issue 入口能降低反馈成本。

**改动范围**：

- `update_manager.py`
- `main.py`
- `settings_dialog.py`
- `README.md`
- `project-log/05-current-status.md`
- `project-log/06-dev-log.md`
- `project-log/12-design-decisions.md`
