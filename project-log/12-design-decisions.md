## 决策 1：不保存网卡字段，实时识别当前优先网卡

**日期**：2026-05-22

**状态**：已采用

**替代关系**：无

**类型**：产品 / 架构

**背景**：

NetSwitch 面向经常切换网络环境的用户。用户可能在家庭 Wi-Fi、公司有线、客户现场网络、VPN 或虚拟网卡之间切换。若方案绑定网卡名，用户换网卡或网卡重命名后容易失效。

**考虑过的方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 用户选择并保存网卡 | 每个方案保存 adapter 字段 | 精确，可控 | 配置复杂，换网卡名后失效 |
| **B. 实时识别当前优先网卡** | 每次切换解析默认路由并查询网卡别名 | 用户无需关心网卡，迁移简单 | 多网卡 / VPN 场景可能误判 |

**最终选择**：

采用方案 B。配置文件不保存 `adapter` 字段，每次切换时通过 `route print 0.0.0.0` + PowerShell 实时识别当前优先网卡。

**依据**：

- 项目核心目标是简单快捷。
- 大多数用户只想切换“当前正在使用的网络”。
- 不保存网卡字段可降低配置迁移成本。

**实现原则**：

- 文档和配置结构中不要新增 adapter 字段，除非后续重新评估此决策。
- 当前实现应改进为真正按 metric 选择默认路由，避免和本决策目标冲突。

**已知不足**：

- VPN、虚拟网卡、多默认路由场景下可能选错。
- 用户无法显式指定“把方案打到某张网卡”。

**后续触发条件**：

- 用户反馈多网卡误判频繁发生。
- 需要支持固定网卡的企业或实验室场景。

**相关文件**：

- `network_controller.py`
- `02-database-design.md`
- `10-planning-log.md` 中的 ADR-001

## 决策 2：运行时托盘状态图标使用 Qt 动态绘制

**日期**：2026-05-22

**状态**：已采用

**替代关系**：无

**类型**：交互 / 架构

**背景**：

托盘图标需要根据网络状态变色：正常、网关不通、切换失败。运行时需要轻量地生成状态图标。

**考虑过的方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 使用多套静态图片 | 为每个状态准备 png/ico | 简单直观 | 资源文件多，状态扩展麻烦 |
| **B. Qt 运行时绘制** | `QPixmap/QPainter` 绘制纯色背景和字母 | 无额外资源，状态切换方便 | 视觉细节不如正式图标 |
| C. Pillow 运行时绘制 | 运行时用 Pillow 生成 | 图像处理能力强 | 增加运行时依赖用途，和 Qt 互转繁琐 |

**最终选择**：

运行时状态图标使用 Qt `QPixmap/QPainter` 动态绘制。Pillow 只在构建期生成 `icon.ico`、`tray_16.png`、`tray_32.png`。

**依据**：

- PyQt6 已是运行时依赖。
- 状态图标只是颜色变化，Qt 绘制足够。
- 减少运行时图标资源管理复杂度。

**实现原则**：

- `tray.py` 中保留状态颜色映射。
- 构建期图标仍由 `scripts/generate_icon.py` 管理。

**已知不足**：

- 当前运行时图标仅绘制纯色背景和白色 N，与构建期图标右下角横线细节不完全一致。

**后续触发条件**：

- 品牌视觉要求运行时托盘图标与正式 ico 完全一致。

**相关文件**：

- `tray.py`
- `scripts/generate_icon.py`

## 决策 3：开机自启同时放在托盘菜单和设置弹窗

**日期**：2026-05-22

**状态**：已采用

**替代关系**：无

**类型**：交互

**背景**：

开机自启是高频设置，用户可能希望不进入设置弹窗即可快速切换。

**考虑过的方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 只放设置弹窗 | 托盘菜单只保留打开主界面、设置、退出 | 菜单更短 | 高频设置多一步 |
| **B. 托盘菜单和设置弹窗都保留** | 托盘菜单直接勾选，设置弹窗也可改 | 方便快速操作 | 同一设置有两个入口，需要保持同步 |

**最终选择**：

采用方案 B。托盘菜单保留「开机自启」，设置弹窗也保留该选项。

**依据**：

- 用户已确认保留托盘菜单「开机自启」更方便。
- 当前代码已实现两个入口同步配置。

**实现原则**：

- 修改任一入口后，都要同步 `profiles.json` 和托盘勾选状态。
- 注册表写入失败不能静默成功，后续需要补充错误反馈。

**已知不足**：

- 当前注册表写入失败被吞掉。
- 当前 Run 路径未显式加引号。

**后续触发条件**：

- 设置项增多导致托盘菜单过长时，重新评估菜单结构。

**相关文件**：

- `main.py`
- `tray.py`
- `settings_dialog.py`

## 决策 4：网络切换等待应从固定等待改为轮询

**日期**：2026-05-22

**状态**：需复核

**替代关系**：无

**类型**：架构 / 交互

**背景**：

当前实现写入配置后固定等待 2 秒。用户确认这是待优化问题，目标是最多等待 5 秒，每 500ms 轮询检测是否生效。

**考虑过的方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 固定 2 秒 | 当前实现 | 简单 | 慢 DHCP 场景不稳 |
| B. 固定 5 秒 | 等更久 | 更稳 | 体验变慢 |
| **C. 最多 5 秒轮询** | 每 500ms 检测一次，生效即继续 | 兼顾速度和可靠性 | 实现略复杂 |

**最终选择**：

目标采用方案 C。当前代码尚未实现，已写入待办和评审问题。

**依据**：

- 网络配置生效时间不固定。
- 工具目标是快速和可靠并重。

**实现原则**：

- 静态 IP 模式应检查目标 IP / 网关是否出现在当前网卡状态中。
- DHCP 模式应等待实际 IP / 网关获取完成。
- 超过 5 秒仍未满足条件，再进入失败或警告处理。

**已知不足**：

- 需要真实 Windows 网络环境验证。

**后续触发条件**：

- 修复 `network_controller.apply_profile()` 时同步完成。

**相关文件**：

- `network_controller.py`
- `10-planning-log.md` 中的 ADR-002
- `11-code-review-log.md`

## 决策 5：更新检查采用 GitHub Release，入口放在设置页

**日期**：2026-05-24

**状态**：已采用

**替代关系**：无

**类型**：产品 / 交互

**背景**：

用户在 Windows 上手动去 Release 下载更新包、卸载旧版再安装新版，流程太长。NetSwitch 需要在软件内提供更直接的更新入口，并同时提供作者和反馈渠道。

**考虑过的方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 启动时自动检查并强提示 | 每次启动都检查更新 | 最自动化 | 容易打扰用户 |
| **B. 设置页手动检查更新** | 在设置页放“检查更新”按钮 | 主动可控，不打扰 | 需要用户点一次 |
| C. 引入独立更新服务 | 后台长期维护更新流程 | 体验完整 | 复杂度高，不适合当前体量 |

**最终选择**：

采用方案 B。设置页提供“检查更新”按钮，发现新版本后可下载 Release 安装包并运行；同时在同一页提供作者信息、GitHub 主页和 Issue 入口。
Windows 更新流程采用“下载官方安装器 → 退出当前 NetSwitch → 运行安装器 → 安装完成后默认启动新版”，不在运行中直接覆盖 exe。

**依据**：

- 更新源固定为 GitHub Release，手动检查已经足够直接。
- 不额外增加常驻后台更新服务，保持工具轻量。
- 设置页是最自然的维护入口，和开机自启等偏好项放在一起也合理。

**实现原则**：

- 更新逻辑放在独立模块，避免和主网络切换逻辑耦合。
- 更新检查失败不能影响当前程序运行。
- 更新检查 / 安装不能打断正在进行的网络切换。
- 只把安装包资产作为软件内更新目标，不能把便携版 exe 当作安装器运行。
- 自动安装更新只对已安装版本开放；便携版用户应打开发布页手动替换便携 exe。
- 下载必须先写临时文件，完整后再替换为可执行安装包。
- 安装器启动失败时不能退出当前程序。
- 安装器完成后默认重新启动 NetSwitch，减少用户手动查找入口。
- 帮助与反馈信息应清晰可见，但不要占用主界面空间。

**已知不足**：

- 目前是手动检查，不是启动即自动提示。
- 下载后仍需用户确认退出并运行安装程序。

**后续触发条件**：

- 后续如果需要更自动的升级体验，再评估是否引入专门更新框架。

**相关文件**：

- `update_manager.py`
- `settings_dialog.py`
- `main.py`
- `README.md`

## 决策 6：软件内自动安装更新必须先校验 Release 安装包完整性

**日期**：2026-05-25

**状态**：已采用

**替代关系**：无

**类型**：安全 / 发布

**背景**：

更新功能会直接下载并运行 GitHub Release 的安装器。若不做完整性校验，软件只是把远端二进制拉到本地再执行，安全边界太弱。

**考虑过的方案**：

| 方案 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| A. 直接下载后运行 | 只校验文件存在 | 实现最简单 | 供应链风险高 |
| **B. 下载后校验 SHA-256** | Release 同时提供安装包和校验文件 | 能确认下载内容未被篡改 | Release 发布流程要多一步 |
| C. 引入签名验证 | 使用代码签名证书或独立签名文件 | 安全性更强 | 成本高，当前体量不适合 |

**最终选择**：

采用方案 B。Release 里上传安装包和对应的 SHA-256 校验文件，软件内自动安装前先比对校验值；若 Release 没有提供校验文件，则不在软件内自动安装，只引导用户打开发布页手动下载。

**依据**：

- 相比“只看文件名”，SHA-256 能有效确认下载内容未被替换。
- 这套方式对 GitHub Release 发布流程改动小，适合当前项目体量。
- 用户仍可保留手动下载入口，不会因为自动安装校验失败而被完全卡住。

**实现原则**：

- 更新检查只接受安装器资产，不接受便携 exe 作为自动安装目标。
- 安装器必须在校验通过后才可拉起。
- Release 没有校验文件时，软件内自动安装应主动退回到发布页手动下载。

**已知不足**：

- 仍然依赖 GitHub Release 资产完整性和发布流程规范。
- 目前没有引入代码签名验证。

**后续触发条件**：

- 若后续要提升到更强的供应链安全，再评估代码签名和更严格的安装包签名验证。

**相关文件**：

- `update_manager.py`
- `.github/workflows/release.yml`
- `main.py`
- `README.md`
